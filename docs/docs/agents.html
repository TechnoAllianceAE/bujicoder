Agent Architecture

BujiCoder uses a multi-agent system where specialized agents collaborate to complete tasks.

How It Works

Instead of one generic chatbot, BujiCoder has a team of purpose-built agents. The orchestrator (base agent) receives your message, decides which specialist to delegate to, and coordinates the workflow.

Each agent has:

  • A system prompt tailored to its role
  • A specific set of tools it can use
  • A model assignment based on the current cost mode

Model Assignment

Agents use models based on their role and the current cost mode:

  • The base (orchestrator) agent uses the main model
  • The file_explorer agent uses the file_explorer model
  • All other agents use the sub_agent model, unless overridden

See Cost Modes for how to configure models per mode and per agent.

Agent Workflow Example

When you ask "Add a login form to the app", the typical workflow is:

  1. base receives the task, spawns planner
  2. planner reads relevant files, creates a step-by-step plan
  3. base spawns editor for each file modification
  4. editor makes precise edits using edit_file/write_file tools
  5. base optionally spawns reviewer to check the changes
  6. base summarises the result for you

Custom Agents

Agent definitions live in YAML files. You can customise existing agents or add your own by placing YAML files in the agents directory:

# Default location
~/.bujicoder/agents/

# Or set a custom path in config
agents_dir: /path/to/my/agents
Note: Use /models in the TUI to see which models are currently assigned to each agent role.

Shared Agent Memory

Agents can share knowledge within a session using shared memory. When a researcher agent discovers something important, it writes to shared memory so the editor agent can read it later.

  • researcher and planner agents have shared_memory_write + shared_memory_read
  • editor has shared_memory_read only (consumes knowledge, doesn't produce research)
  • The shared memory summary is injected into every agent's system prompt automatically

See Project Memory for full details on both persistent and session memory.

Context Compaction

During long-running agent tasks, BujiCoder automatically compacts conversation history when approaching the model's context window limit (70% threshold). It uses a lightweight model to summarize older messages while keeping the most recent 8 messages verbatim. If the LLM summarization fails, it falls back to extractive truncation.

Max Mode — Parallel Evolution

In Max mode, when editing files, the parallel_editor agent can spawn multiple competing implementations simultaneously. A judge evaluates all proposals and picks the best one, resulting in higher quality output at the cost of more API calls.
