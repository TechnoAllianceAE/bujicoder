Command Security

BujiCoder v0.8.4 introduces a 4-level threat classification system that analyses every shell command before execution, preventing destructive operations from running unchecked.

Overview

When an agent calls run_terminal_command, BujiCoder parses the command string, splits it into individual statements, and assigns each a threat level. The threat level determines whether the command runs immediately, requires user approval, or is blocked outright.

This system protects against both direct destructive commands and indirect execution techniques such as eval, $(), and piped installers.

Threat Levels

LevelDescriptionExamples
NoneSafe, read-only commandsls, git status, cat file.txt, pwd
LowMildly risky, local side effectsnpm install (local), go build
MediumPotentially destructive, reversiblerm file.txt, git push, chmod 755, docker rm
HighDestructive, difficult to reverserm -rf dir/, git push --force, git reset --hard, kill -9, DROP TABLE
CriticalSystem-destroying, unrecoverablemkfs, dd if=... of=/dev/, rm -rf /, fork bombs

What Gets Detected

The classifier checks each command statement against the following categories:

CategoryDetected Patterns
Meta-execution$(), backticks, eval, bash -c, sh -c — prevents bypass via indirect execution
Filesystem destructionrm -rf, recursive deletes, rm -r
Git destructive opsgit push --force, git reset --hard, git clean -f, git branch -D
SQL destructiveDROP TABLE, TRUNCATE, DELETE without WHERE
Process killingkill -9, pkill, killall
System servicessystemctl stop, systemctl restart
Sensitive paths~/.ssh, ~/.aws, /etc/shadow, and other credential stores
Remote code executioncurl | sh, wget | bash, and similar piped installers
find -execfind ... -exec with destructive sub-commands

Permission Modes

The permission mode controls how threat levels translate into action. Set the mode in ~/.bujicoder/permissions.yaml or via the /permissions slash command.

ModeNone / LowMediumHighCritical
strictAuto-approveBlockedBlockedBlocked
ask (default)Auto-approvePrompt userPrompt userBlocked
yoloAuto-approveAuto-approveAuto-approveBlocked
Note: Critical-level commands are always blocked regardless of the permission mode. There is no override for commands classified as Critical.

permissions.yaml

You can define per-command overrides in ~/.bujicoder/permissions.yaml to allow or deny specific patterns without changing the global mode:

mode: ask
commands:
  - pattern: "go test*"
    action: allow
  - pattern: "git push --force*"
    action: deny

Rules are evaluated top-to-bottom. The first matching pattern wins. If no rule matches, the global mode applies.

  • allow — auto-approve the command without prompting, regardless of threat level
  • deny — block the command outright, regardless of threat level
Important: Even an allow rule cannot override a Critical-level block. Critical commands are unconditionally blocked by the classifier before permission rules are evaluated.

Quote-Aware Parsing

BujiCoder splits compound commands on |, &&, ||, ;, and newlines to classify each statement independently. The parser respects quoted strings, so content inside single or double quotes is not treated as a command separator.

# Correctly treated as a SINGLE command (no split on the semicolon):
echo "hello; world"

# Correctly treated as TWO commands (split on the unquoted semicolon):
echo hello; rm -rf /tmp/build

This prevents false positives where string literals contain characters that look like shell operators.
