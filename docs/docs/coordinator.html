Goal Coordinator

The /goal command decomposes high-level goals into a task DAG (directed acyclic graph), executes tasks in dependency order with specialist agents, and synthesizes the results into a coherent response.

Usage

Invoke the goal coordinator by typing /goal followed by a description of what you want to accomplish:

/goal Add authentication with JWT login and protected routes
/goal Refactor the database layer to use connection pooling
/goal Set up CI/CD with GitHub Actions for this Go project
Tip: The /goal command works best with complex, multi-step tasks. For simple single-file edits, just describe what you want directly.

How It Works

When you run /goal, BujiCoder follows a structured pipeline to break down and execute your request:

  1. The planner agent analyzes your goal and generates a JSON task plan.
  2. Each task specifies: id, title, description, agent_id, and depends_on (optional).
  3. Tasks with no dependencies start immediately in parallel.
  4. Tasks with dependencies wait for their prerequisites to complete.
  5. Results from completed prerequisites are injected as context into dependent tasks.
  6. All tasks share memory via the shared agent memory system.
  7. After all tasks complete, a synthesis step combines outputs into a coherent response.

Available Agents for Tasks

The planner assigns each task to a specialist agent based on the nature of the work:

AgentPurpose
researcherRead-only investigation of code, docs, web
editorFile editing and code changes
file_explorerFast, lightweight file discovery
plannerFurther task decomposition
reviewerCode review for correctness and style
thinkerDeep reasoning about complex problems
git_committerGit operations with proper commits

Dependency Resolution

The coordinator manages task execution order using the dependency graph:

  • Parallel execution: Independent tasks run concurrently, up to 6 at a time.
  • Cycle detection: The coordinator validates the task graph before execution and rejects plans with circular dependencies.
  • Cascade failure: If a task fails, all downstream dependents are automatically marked as failed.
  • Plan size limit: A maximum of 20 tasks per goal prevents runaway plans.

Example Task Plan

For a goal like "Add authentication with JWT login and protected routes", the planner might generate:

[
  {"id": "t1", "title": "Research current auth setup", "agent_id": "researcher", "depends_on": []},
  {"id": "t2", "title": "Plan JWT implementation", "agent_id": "planner", "depends_on": ["t1"]},
  {"id": "t3", "title": "Implement JWT middleware", "agent_id": "editor", "depends_on": ["t2"]},
  {"id": "t4", "title": "Add login endpoint", "agent_id": "editor", "depends_on": ["t2"]},
  {"id": "t5", "title": "Review changes", "agent_id": "reviewer", "depends_on": ["t3", "t4"]}
]

In this plan, t3 and t4 run in parallel since they both only depend on t2. The reviewer in t5 waits for both editing tasks to finish before inspecting the combined changes.

Context Compaction

During long-running agent tasks, BujiCoder automatically compacts conversation history when approaching context window limits. This uses a lightweight model to summarize older messages while keeping recent context verbatim, ensuring that even large multi-task goals complete without running out of context.
