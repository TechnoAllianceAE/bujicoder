Project Memory

BujiCoder has two memory systems: persistent project memory that survives across sessions, and shared agent memory for inter-agent communication within a single session.

Persistent Project Memory (BUJI.md)

Persistent memory lets agents store and recall knowledge across sessions. Architecture decisions, naming conventions, common pitfalls, and project-specific patterns all survive between runs so the agent does not have to re-discover them each time.

How It Works

Memory is stored in a markdown file called BUJI.md. At the start of every session, BujiCoder loads this file and injects its contents into the system prompt so all agents have immediate access to the accumulated knowledge.

Content is organized under ## Section Name headers. Agents write to specific sections, and existing sections are updated in place rather than appended to.

File Location

BujiCoder looks for the memory file in two locations, in order:

  1. .bujicoder/BUJI.md — preferred location, read-write
  2. BUJI.md in the project root — read-only fallback

If neither file exists, BujiCoder creates .bujicoder/BUJI.md automatically the first time an agent writes to memory.

Tip: You can manually create a BUJI.md in your project root to pre-seed memory before the first session.

Tools

ToolPurpose
memory_readReads the contents of BUJI.md
memory_writeWrites content to a named section in BUJI.md

Slash Command

Use /memory in the TUI to view the current project memory without consuming an LLM call.

Limits

  • System prompt injection: capped at 6KB. If BUJI.md exceeds this, it is truncated when loaded into the prompt.
  • File size: capped at 32KB to prevent unbounded growth. Agents will refuse to write beyond this limit.

What Gets Stored

The base agent has built-in guidance on when to write to persistent memory. Common entries include:

  • Architecture decisions and rationale
  • Naming conventions and code style preferences
  • Common pitfalls discovered during sessions
  • Project-specific patterns and idioms

Shared Agent Memory (Session-Only)

Shared memory is an ephemeral key-value store that lets agents exchange information within a single session. When the session ends, all shared memory is discarded.

How It Works

The store is a thread-safe, namespaced key-value map. Each agent writes under its own namespace using the pattern agentID/key, preventing collisions between agents.

A markdown summary of all shared memory entries is injected into every agent's system prompt, so any agent can read what other agents have written without making an explicit tool call.

Tools

ToolAvailable ToPurpose
shared_memory_writeresearcher, plannerWrite a key-value pair to the shared store
shared_memory_readeditorRead entries from the shared store

Use Case

A typical workflow:

  1. The researcher agent discovers the project's API structure and writes it to shared memory
  2. The planner agent reads the research findings and writes a step-by-step plan
  3. The editor agent reads both the API structure and the plan when making file changes

Limits

  • Max entries: 200 key-value pairs across all namespaces
  • Value size: 10KB per individual value
  • Summary size: 8KB total for the markdown summary injected into prompts
